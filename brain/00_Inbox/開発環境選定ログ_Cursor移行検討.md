# 開発環境の再構築・選定ログ

作成日: 2026-03-29

## 1. 現状の課題と「手放したい」不満

現在、**VS Code + Claude Code** で開発を行っているが、以下のジレンマに直面している。

- **管理コストの肥大化:** 自由度が高い反面、AIがプロジェクト構造を正確に守るための「お膳立て（設定や指示）」を人間が細かく行う必要があり、本業以外の「AIの世話」に時間を取られている
- **秩序の崩壊:** エージェント（Claude Code）が独断でフォルダを生成したり、既存ルールを無視したりすることで、後から人間が「交通整理」をする手間が発生している
- **ツールの分断によるノイズ:** 発見（Gemini）と実行（Claude）を別環境で行う際のスイッチングコストが、集中力を削いでいる

## 2. 意思決定の指針：AIへの「権限委任」

今回の環境移行の核となる思想は、**「重要ではない部分のコントロールをAIに手放し、本業に専念する」**こと。

- **「自由」より「秩序」:** 1から10まで指示して動かす自由度よりも、一定の制約（ルール）の中でAIが自律的にフォルダ管理や整合性を維持してくれる「楽さ」を優先する
- **不満が出た際の逃げ道:** まずはAI統合型で効率を最大化し、もし自由度が足りないと感じるほど習熟・成熟した際には、再度VS Codeのようなフルコントロール環境へ戻るという「段階的な構え」を取る

## 3. 主要なAI IDE環境の比較とCursor選定理由

### VS Code + 拡張
- 自由度は最高だが、AIとの統合が「後付け」で環境維持の労力が高い
- **判断: 保留** — 今は管理コストを下げたい時期

### Windsurf
- 「Flow」という強力なエージェント機能を持ち、Cursorに近い
- **判断: 比較** — ユーザー数・知見・安定性でCursorを優先

### GitHub Copilot Workspace
- GitHub連携は強いが、特定のワークフローに縛られがち
- **判断: 比較** — ローカル環境でのGemini/Claudeスイッチがしにくい

### Cursor ← 採用
- エディタ自体がAI。インデックス化が標準で、フォルダ構成の維持能力が高い
- 思考（Gemini）と実行（Claude）のスイッチが最もシームレス
- **判断: 採用**

**Cursorを選んだ決め手：**
「AIがプロジェクトの全体像（地図）を常に持っている」という安心感。人間がいちいちフォルダを指定しなくても、AIが自律的に適切な場所へアウトプットしてくれるため、**「重要じゃない作業を切り捨てる」**という意図に最も合致する。

## 4. 今後の運用マニュアル（案）

1. **「発見」フェーズ（Gemini）:** Cursor内でGeminiを使い、認知外の突破口や検索結果を得る
2. **「委任」フェーズ（Claude/Composer）:** Geminiとの会話を引き継ぎ、Claudeに「この方針で、フォルダ構成を守りつつ実装して」と丸投げする
3. **「監視」フェーズ:** `.cursorrules` を「現場監督」として機能させ、AIが勝手な振る舞いをしないよう最低限のガードレールだけ敷いておく

## 5. Cursorの歴史と「なぜ今なのか」

Cursorは思ったより歴史がある。YouTubeで去年夏（2025年夏）ごろの動画が多いのは、その時期に機能が「魔法レベル」に達したからであって、誕生自体はもっと前。

### 歩みのターニングポイント

- **2023年初頭：** VS Codeのフォークとして誕生。「GitHub Copilotが使いやすいエディタ」程度の認識だった
- **2024年：** プロジェクト全体をAIが検索可能にする「Codebase Indexing」を確立。複数ファイルをまたいだ指示が通るようになった
- **2025年夏（爆発的普及）：** Claude 3.5 SonnetとComposer（複数ファイルを同時書き換え）が組み合わさり、「次元が違う」体験が生まれた
- **2026年現在：** ターミナル操作やエラー修正を自ら行う「Agent Mode」搭載。単なるエディタを超えた

### 「本物のエンジニア」が何を使っているか

トレンドではなく「実利」で選んでいる層の動向：

- **超効率重視のシニア / スタートアップ → Cursor**
  「フォルダ管理や定型作業に脳のリソースを使いたくない。価値を生むロジック設計に全振りしたい」
- **環境構築・検証マニア / インフラ寄り → VS Code + Claude Code / Roo Code**
  「AIの挙動を1から10まで把握し、完全にカスタマイズしたい。ブラックボックスは嫌だ」という職人気質層
- **エンタープライズ（大手）→ GitHub Copilot / VS Code**
  セキュリティやガバナンスの観点から、枯れた技術と標準ツールを使い続ける必要がある層

NVIDIAのような世界トップクラスのエンジニア集団（約3万人）が全社的にCursorを導入した事例もある。「楽をするため」ではなく「より高度な問題に集中するため」に選んでいる。

### 「ゴリゴリ開発」の定義が変わった

- **昔：** 複雑な正規表現を暗記し、ディレクトリ構造を完璧に管理し、タイピング速度で圧倒する
- **今：** 抽象的な設計図をAIに示し、自分はエッジケースやセキュリティの脆弱性チェックに全神経を注ぐ

「手放す」のは技術力の低下ではなく、**管理という名の雑務からの解放**。

## 6. 現実的な制約：一人で営業・開発・リリースをやる

- **やっていること:** 営業 → 受注 → システム開発 → リリース を一人で回す
- **本音:** ツールの複雑さを制御できてる感覚は好き。でもそこに踏み込みすぎると本業が回らない
- **気づき:** 「いい感じのファイル構成」への要求水準は時間とともに上がっていく。だから今のうちにCursorに任せる方向に舵を切る方が、長期的に楽になれる可能性がある

→ **結論:** 「切り捨て」は妥協ではなく、一人で事業を動かすための合理的な選択
